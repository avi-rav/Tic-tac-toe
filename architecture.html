Architecture Reference

Tic-Tac-Toe

A React + TypeScript game built around a strict separation between pure rules, state ownership, and presentation. Play two-player hot-seat or vs. an AI opponent at three difficulties. Win detection, turn order, and move selection are all data-driven; persistence and validation each live behind a single seam. This page maps every component, hook, and module in the codebase.

React 19 TypeScript 6 Vite 8 Vitest + Testing Library Playwright e2e react-hook-form + zod CSS Modules

All under src/components/. Each is presentational — it receives data and emits intents through callbacks, owning no rule logic. Every component ships a matching *.module.css.

PlayerSetup view

components/PlayerSetup/

react-hook-form + zodResolver. Picks a mode (Two Players / vs Computer); collects names, or the human's name + a difficulty. Validates against playerSchema, hands the players and chosen opponent off on submit.

Game view

components/Game/

Composition root for a session. Pulls state from useGame() and wires it to children. When the opponent is an AI, a cleanup-guarded setTimeout effect feeds aiStrategy.chooseMove into the same makeMove seam (board disabled meanwhile).   • opponent: Opponent
  • onChangePlayers(): void

03 Hooks — the state APIs

These are the app's internal APIs: the only places that own mutable state. Components depend on these interfaces (Dependency Inversion), not on the rules or localStorage directly.

useGame() hook

src/hooks/useGame.ts

Single owner of all mutable game state. Status and winner are derived from the board (via useMemo), never stored separately, so they can't fall out of sync.

Returns — UseGameResult
  • board: BoardState
  • currentPlayer: Player
  • status: GameStatus
  • winner: Player | null
  • winningLine: number[] | null
  • scores: Scores
  • makeMove(i: number): void
  • newGame(): void
  • resetScores(): void

useHistory() hook

src/hooks/useHistory.ts

Owns the in-memory history list, seeded once from storage via a lazy initializer. Delegates all persistence to historyStorage.

Returns — UseHistoryResult
  • history: GameRecord[]
  • addRecord(r: GameRecord): void
  • clear(): void

04 Domain core & data seams

Framework-free modules. Pure functions and data only — trivially unit-tested.

gameLogic.ts pure

src/game/gameLogic.ts

Deterministic rules driven entirely by WINNING_LINES / PLAYERS.

Functions
  • createEmptyBoard(): BoardState
  • calculateWinner(b): WinResult | null
  • isDraw(b): boolean
  • getNextPlayer(p): Player

aiStrategy.ts pure

src/game/aiStrategy.ts

Pure move selection. Tic-tac-toe is solved, so difficulty = how often the AI plays sub-optimally: easy random, medium win/block heuristic, hard depth-aware minimax (unbeatable). The rng is injectable for deterministic tests.

Exports
  • Difficulty = 'easy'|'medium'|'hard'
  • chooseMove(b, p, d, rng?): number | null

constants.ts config

src/game/constants.ts

Open/Closed config: extend the game by editing data here, not branching logic.

Exports
  • BOARD_SIZE = 9
  • PLAYERS = ['X','O']
  • WINNING_LINES: number[][]

playerSchema.ts validation

src/game/playerSchema.ts

Single source of validation truth (zod) — a discriminatedUnion on mode: two-players needs two distinct names; vs-computer needs the human's name + a difficulty. Names ≥ 2 chars, trimmed. Form value type is inferred so rules & types can't drift.

Exports
  • playerSchema
  • PlayerFormValues (inferred)

historyStorage.ts persistence

src/history/historyStorage.ts

The only module that touches localStorage. Every access try/catch-wrapped so a disabled/full/corrupt store degrades to an empty list. Caps at MAX_HISTORY = 20, newest-first.

Functions
  • loadHistory(): GameRecord[]
  • saveRecord(r): GameRecord[]
  • clearHistory(): void

game/types.ts types

src/game/types.ts

Shared domain types. Player is derived from PLAYERS so adding a marker never edits the type.

Types
  • Player · CellValue · BoardState
  • GameStatus · Players · Scores
  • WinResult · Opponent

history/types.ts types

src/history/types.ts

The persisted record shape — a snapshot enough to redraw the final board without storing an image.

Exports

05 Key data flow — a move

  • 1. User clicks a Square → Board.onCellClick(i) → Game → useGame.makeMove(i)
  • 2. makeMove guards illegal moves, writes the new board, then asks calculateWinner / isDraw on the resulting board to update scores or flip the turn
  • 3. status & winner are re-derived (useMemo) → flow back down to StatusBar, ScoreBoard, Board
  • 4. vs. AI: if it's now the AI's turn, Game's effect calls aiStrategy.chooseMove and schedules makeMove(i) (setTimeout, board disabled) — looping back through step 2. Cleanup cancels the timer, so StrictMode never double-moves.
  • 5. On a terminal state, Game's effect fires once → onGameEnd → useHistory.addRecord → historyStorage.saveRecord → localStorage
